# 双代理协作底座设计

> 仓库：`dreambuddy-v1`  
> 版本：v1  
> 日期：2026-05-16  
> 状态：legacy（历史参考；当前以 `agent-ledger-protocol-vs-governance-short-spec.md` 与 `agent-efficient-collaboration-mode.md` 为准）  
> 目标：记录早期“双 AGENT 并行开发”协作底座的设计取舍，作为多 AGENT 协作的子集参考，优先服务 `7-ARTIFACT-HUB-V2` 的实现阶段。

## 0. 背景

当前 `7-ARTIFACT-HUB-V2` 已完成一轮以治理中枢为核心的设计文档建设，文档层已经明确：

- 研究中台、市场化中台、交易链路监控、治理控制台、董事会总览台的目标结构；
- `/feed`、`/chain` 属于遗留实现基线；
- `ops-ui`、市场化中台、董事会总览台仍属于规划中页面；
- 后续实现需要同时覆盖页面、Hub 服务、治理契约、工作流与审计链路。

在这种上下文下，如果直接让两个 AI 代理同时开始编码，极易出现以下问题：

- 同时修改共享文件，导致互相覆盖；
- 前后端边界未冻结，导致并行开发后无法接线；
- 文档、契约、实现混在一起推进，PR 很快失控；
- 用户被迫介入大量微观技术决策，失去“只看关键里程碑”的体验。

因此第一阶段不直接做功能，而是先建立一套“协作底座”。

## 1. 目标

本设计的目标不是马上实现某个页面，而是先为后续实现提供稳定协作机制。

第一阶段的目标是：

- 明确双代理与用户之间的角色和决策边界；
- 明确任务拆分方式与主责域；
- 冻结共享契约与共享文件的修改规则；
- 建立分支、里程碑、联调和冲突处理机制；
- 为后续 `ops-ui`、`/feed`、`/chain` 的并行开发提供可复用工作流。

本设计的成功标准是：

- 两个代理可以并行推进，不频繁改同一批文件；
- 前后端可以围绕冻结契约并行开发；
- 用户只需要在关键里程碑介入，而不需要参与日常技术裁决；
- 第一轮里程碑结束后，可以自然切换到具体功能开发。

## 2. 适用范围

本方案优先服务以下场景：

- `7-ARTIFACT-HUB-V2` 的规划功能逐步落地；
- `ops-ui`、`/feed`、`/chain` 等跨前后端模块的并行开发；
- 文档、契约、后端、前端需要同时演进，但不希望互相阻塞。

本方案暂不覆盖：

- 多于两个 AI 代理的复杂自治协作；
- 自动化任务调度平台；
- 完整项目管理系统或自动派单系统；
- 与外部 CI/CD 平台的高级集成策略。

## 3. 参与角色

本方案基于 3 人模型：

- 用户
- SOLO
- Claude Code

### 3.1 用户角色

用户不参与日常实现细节，而是负责：

- 确认方向与阶段目标；
- 查看关键里程碑结果；
- 批准目标变更、模块边界变更、是否进入下一阶段。

### 3.2 SOLO 角色

SOLO 作为架构与集成主控，主责包括：

- 协作规则与任务拆分；
- 接口契约与共享类型；
- 治理规则、schema、后端能力与集成收口；
- 风险识别、冲突裁决与里程碑整理；
- README、spec、共享文档收口。

### 3.3 Claude Code 角色

Claude Code 作为页面与模块实现主力，主责包括：

- 页面壳与交互层；
- 基于冻结契约的前端接入；
- 局部模块实现与可视化工作流页面；
- 在不修改协作底座前提下的快速产出。

## 4. 协作模式选择

本设计比较过三种模式：

### 4.1 单主控代理

- SOLO 拆任务并主导全部集成；
- Claude Code 作为执行副手。

优点：

- 冲突最少；
- 路径最稳定。

缺点：

- Claude Code 被串行化；
- 自治度不足，不符合当前目标。

### 4.2 双代理分域自治

- SOLO 与 Claude Code 各自主责清晰模块；
- 通过冻结契约、共享文件规则和里程碑集成来并行推进。

优点：

- 最符合“双代理自治更多”的目标；
- 速度与稳定性平衡较好；
- 最适合当前 `7-ARTIFACT-HUB-V2` 这种可拆前后端与治理边界的项目。

缺点：

- 前期必须先建立协作底座；
- 若契约漂移，返工成本会升高。

### 4.3 接力式自治

- SOLO 先定设计与骨架；
- Claude Code 接着做实现；
- SOLO 再回收集成。

优点：

- 比单主控快；
- 比完全并行稳。

缺点：

- 并行度不足；
- 更像高效接力，而不是真正并行。

### 4.4 结论

推荐采用：

> 双代理分域自治

原因是它最符合当前用户目标，也最接近行业中“平台/后端先定边界，前端按契约并行接入”的经典做法。

## 5. 主责域划分

### 5.1 SOLO 主责域

默认由 SOLO 主责：

- `7-ARTIFACT-HUB-V2/src/**`
- `dreambuddy/meta/**`
- `dreambuddy/config/**`
- 治理规则、对象模型、接口契约、共享 schema
- 协作底座文档与里程碑集成文档

### 5.2 Claude Code 主责域

默认由 Claude Code 主责：

- 页面层与交互层
- `ops-ui` 页面实现
- `/feed`、`/chain` 的前端入口与可视化部分
- mock adapter、real adapter 接入层
- 基于冻结契约的 UI 状态与组件组织

### 5.3 共享但需先申请的文件

以下文件默认不允许双方随意同时修改：

- 顶层 `README`
- 顶层 spec 与 milestone index
- 跨模块公共 `types`
- 公共 `schema`
- 顶层页面入口

对这些文件，应采用“占用权”机制：

- 一个时段只允许一个主责代理修改；
- 另一方如需变更，应先提出需求，再由主责代理收口。

## 6. 任务拆分机制

所有任务先分成三类：

### 6.1 A 类：协作底座任务

包括：

- 接口契约
- schema 定义
- 目录边界
- 共享类型
- 联调规则

默认由 SOLO 主责。

### 6.2 B 类：后端 / 治理任务

包括：

- Hub 服务
- route / trace / audit
- workflow / meta / artifact 接口
- fail-closed / approval / proposal 相关能力

默认由 SOLO 主责。

### 6.3 C 类：前端 / 页面任务

包括：

- `ops-ui`
- `/feed`
- `/chain`
- 页面状态管理
- 页面组件与交互

默认由 Claude Code 主责。

### 6.4 并行条件

一个任务只有同时满足以下 3 条，才允许并行：

- 文件边界清楚；
- 输入输出清楚；
- 不依赖对方当天改完同一文件。

若不满足这 3 条，则该任务不并行，改为接力推进。

### 6.5 任务卡模板

建议所有任务统一使用以下字段：

- `任务名`
- `目标`
- `主责代理`
- `输入依赖`
- `输出产物`
- `禁止修改范围`
- `完成判定`

## 7. 分支与里程碑规则

### 7.1 三层分支模型

建议采用以下分支结构：

- `main`
- `milestone/*`
- `agent/*`

### 7.2 `main`

`main` 只接收经过里程碑验证的稳定内容。

### 7.3 `milestone/*`

每个阶段建立一个里程碑集成分支，例如：

- `milestone/dual-agent-foundation`
- `milestone/ops-ui-foundation`
- `milestone/feed-rebuild`
- `milestone/chain-dual-workflow`

该分支默认由 SOLO 负责收口与集成。

### 7.4 `agent/*`

双代理各自使用独立工作分支，例如：

- `agent/solo/ops-contracts`
- `agent/solo/governance-schema`
- `agent/claude/ops-ui-shell`
- `agent/claude/chain-visual-panel`

规则如下：

- Claude Code 不直接写 SOLO 分支；
- SOLO 不直接在 Claude Code 的实现分支上做大改；
- 双方先提交到各自 `agent/*`；
- 再由 SOLO 收口到 `milestone/*`；
- 里程碑稳定后再合到 `main`。

## 8. 契约冻结机制

并行开发的核心前提不是“谁更快”，而是“接口是否先冻结”。

这里的接口包括：

- HTTP API
- TypeScript type
- JSON schema
- frontmatter 字段
- task/result 文件结构
- 页面消费的数据 shape
- 目录与命名约定

### 8.1 冻结顺序

建议按以下顺序冻结：

1. 最小字段
2. 读接口
3. 写接口

也就是先保证页面可读可展示，再逐步进入高风险写动作。

### 8.2 冻结等级

- `L0 临时接口`
  - 仅允许单代理本地试验；
  - 不能作为双代理协作依据。
- `L1 协作接口`
  - 已写入 spec 或 type 文件；
  - 可供双代理并行依赖。
- `L2 里程碑接口`
  - 已完成一轮联调；
  - 可作为后续子任务默认依赖。

只有 `L1` 及以上，双代理才能并行依赖。

## 9. 共享文件规则

共享文件分为三类：

### 9.1 契约文件

包括：

- `types`
- `schema`
- API contract
- frontmatter 规范

默认由 SOLO 主责。

### 9.2 入口文件

包括：

- 路由入口
- 顶层页面入口
- 服务启动入口

由模块主责代理修改；如会影响另一代理任务，必须先冻结后修改。

### 9.3 汇总文件

包括：

- README
- spec index
- milestone checklist

默认由 SOLO 收口，避免双方高频改动导致文档漂移。

## 10. 联调方式

建议采用三段式联调：

### 10.1 假数据联调

- Claude Code 先基于冻结的 type / schema / mock data 起页面；
- SOLO 不必等待全部后端实现完成。

目标是先证明页面骨架和交互边界成立。

### 10.2 真实接口联调

- SOLO 提供最小可用真实接口；
- Claude Code 把 mock 切换到真实数据源。

目标是验证“字段一致、能接通、能展示”。

### 10.3 集成联调

由 SOLO 统一检查：

- 字段是否漂移；
- 错误态是否一致；
- 空态、缺字段、失败态是否兜底；
- 页面是否误依赖未冻结字段。

### 10.4 联调产物

每个里程碑至少要有四类产物：

- `contract`
- `mock`
- `implementation`
- `integration note`

## 11. 里程碑定义

建议按“可验证能力”划里程碑，而不是按页面名称硬切。

### M1：协作底座

包括：

- 协作规则文档
- 分支规则
- 任务卡模板
- 契约冻结规则
- 共享文件占用规则
- 第一批最小契约清单

### M2：ops-ui 骨架可联调

包括：

- 页面壳
- 最小健康接口
- trace 查询契约
- mock -> real adapter 切换路径

### M3：`/feed` 恢复为遗留基线可用

包括：

- 基础列表
- 详情页
- 基础数据映射

### M4：`/chain` 双工作流最小展示

包括：

- legacy chain
- `workflow_id / workflow_type`
- 双工作流可视化占位

## 12. 冲突处理机制

双代理自治不意味着无冲突，而是要把冲突处理标准化。

### 12.1 文件冲突

表现为：

- 双方同时改同一共享文件；
- 或一方修改影响另一方未完成任务。

处理方式：

- 暂停该共享文件上的并行开发；
- 由 SOLO 统一回收并重新切边界。

### 12.2 接口冲突

表现为：

- 前端依赖字段与后端契约不一致；
- 双方对数据 shape 理解不同。

处理方式：

- 以冻结契约为准；
- 不以任一实现为准。

### 12.3 目标冲突

表现为：

- 一方追求“快速可用”；
- 一方追求“长期正确”。

处理方式：

- 优先保证当前里程碑目标；
- 长期方案写入下一里程碑，不在当前任务中临时扩 scope。

### 12.4 裁决规则

- 小冲突：SOLO 直接裁决并更新协作底座
- 中冲突：SOLO 给出两个修法，由用户看差异后拍板
- 大冲突：暂停并行，退回上一个冻结点，重新拆任务

## 13. PR 评论协作协议

除了本地门禁与分支规则，双代理还必须通过 PR 评论公开广播任务意图。

这样做的目的有三点：

- 让另一代理在提交前就知道当前谁正在做什么；
- 让用户能直接通过 PR 评论看到当前占用范围；
- 弥补 `conflict gate` 无法感知“对方准备做但尚未提交”的盲区。

### 13.1 强制流程

每次任务必须遵守以下顺序：

1. 先运行 `dual-agent-conflict-gate`
2. 若结果为 `SAFE` 或 `WARNING`，先发 `STARTED` 评论，再开始动手
3. 若任务范围发生变化，发 `UPDATED` 评论
4. 若门禁输出 `BLOCK` 或任务中途受阻，发 `BLOCKED` 评论
5. 任务完成并提交后，发 `DONE` 评论

### 13.2 评论状态词

统一使用以下状态词：

- `STARTED`
- `UPDATED`
- `BLOCKED`
- `DONE`

### 13.3 STARTED 模板

```md
[协作开工声明 / STARTED]

Agent: <agent_id>
任务: <任务名称>
分支: <agent/* 分支>
计划修改:
- <文件或目录 1>
- <文件或目录 2>

预期产出:
- <产出 1>
- <产出 2>

占用范围:
- <当前请勿并行修改的文件或目录>

冲突门禁结果:
- decision: SAFE | WARNING
- reason_codes: <如无可写 []>

状态: STARTED
说明: 在我发 DONE 评论前，请不要并行修改上述范围。
```

### 13.4 UPDATED / BLOCKED / DONE 模板

```md
[协作状态更新 / UPDATED]

Agent: <agent_id>
任务: <任务名称>
变更说明:
- <新增范围或缩减范围>
当前占用范围:
- <更新后的文件或目录>
状态: UPDATED
```

```md
[协作阻塞通知 / BLOCKED]

Agent: <agent_id>
任务: <任务名称>
阻塞原因:
- <门禁 BLOCK 或中途发现的冲突>
需要对方配合:
- <需要谁先完成什么>
状态: BLOCKED
```

```md
[协作完成回报 / DONE]

Agent: <agent_id>
任务: <任务名称>
提交: <commit sha>
已完成:
- <完成项 1>
- <完成项 2>
状态: DONE
说明: 另一代理可以基于最新 PR head 继续。
```

### 13.5 适用范围

本协议适用于（历史试点示例）：

- 任意采用本协议的协作 PR（示例：PR9）；
- 后续 `milestone/*` 集成分支；
- 所有共享文件、共享契约、共享页面入口相关任务。

## 14. 第一轮启动建议

建议第一轮不直接实现具体功能，而是先启动：

> `M1: dual-agent-collaboration-foundation`

### 14.1 M1 完成标准

- 有正式协作文档；
- 有分支规则；
- 有任务卡模板；
- 有最小契约清单；
- 有里程碑 checklist；
- 有首批任务分派。

### 14.2 SOLO 首批任务

- 编写协作底座 spec；
- 固化分支规则、任务卡模板、冲突处理规则；
- 定义第一批最小契约：
  - `health`
  - `trace summary`
  - `route decision summary`
  - `workflow summary`
- 产出 mock contract / type 草案；
- 建立第一个里程碑分支。

### 14.3 Claude Code 首批任务

- 基于冻结的 mock contract，准备页面骨架开发模式；
- 不直接修改共享契约文件；
- 搭建可替换数据源的页面壳模式：
  - mock adapter
  - real adapter 占位
  - 页面状态边界

### 14.4 用户只需关注的关键里程碑

用户只需看四个点：

- `K1 协作底座已写完`
- `K2 最小契约已冻结`
- `K3 双代理已完成第一次并行且无覆盖冲突`
- `K4 第一个功能里程碑准备启动`

## 15. 风险与约束

### 15.1 最大风险

最大风险不是开发速度，而是边界漂移：

- 契约边界漂移；
- 模块责任边界漂移；
- 共享文件无人收口；
- 双代理都以为对方会处理集成问题。

### 15.2 应对原则

应对方式不是增加审批层，而是：

- 先冻结最小契约；
- 先切清主责域；
- 用里程碑分支收口；
- 用 checklist 而不是口头同步保证一致性。

## 16. 一句话结论

这套协作底座的核心不是“让两个 AI 一起写代码”，而是：

> 先冻结边界，再让双代理在清晰主责域内自治推进，最后通过里程碑分支统一集成，让用户只在关键节点介入。
