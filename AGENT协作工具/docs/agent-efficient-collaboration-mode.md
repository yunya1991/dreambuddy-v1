# AGENT 高效协作模式

> 版本：v1 草案  
> 日期：2026-05-17  
> 目标：在不削弱协作边界与可审计性的前提下，把双代理协作从“默认串行接力”调整为“默认并行开发、例外同步收口”的高吞吐执行模式。

## 1. 背景

现有 `AGENT` 协作体系已经建立了冲突门禁、结构化评论、非主责评审、测试证据与合入监督机制。

这些规则提高了过程完整性，但在实际执行中暴露出一个明显问题：

- 原本可以并行推进的子任务，被过早串成“等对方评论 -> 等对方评审 -> 再继续”的接力链；
- 许多确定性小修复也必须先停下来沟通，再交还原 owner 处理；
- 共享计划文件、阶段评论与 PR 评审被高频触发，导致吞吐下降。

因此需要为现有生命周期补充一套默认执行策略，把“严格同步”收敛到真正高风险的节点，把“默认并行”作为常规模式。

## 2. 设计目标

本模式的目标不是放松边界，而是重新分配同步成本。

核心目标如下：

- 默认允许 owner 在已声明目录内并行开发，不要求每个小步骤逐次等待；
- 允许对确定性、小范围、低歧义的问题进行直接接管修复；
- 仅在共享文件、冻结契约、主分支合并、边界变更等关键节点强制同步；
- 把 review 从“逐步 review”改为“阶段 review”与“批量收口 review”；
- 保持现有 `dual-agent-conflict-gate`、`STARTED / UPDATED / BLOCKED / DONE`、`DESIGN_REVIEW`、`TEST_REPORT` 与监督规则兼容。

## 3. 核心原则

### 3.1 默认并行开发

只要任务位于已分配的 owner 目录内，并且不触碰共享文件、冻结契约或共享入口，就应默认并行推进。

这意味着：

- owner 不必为每个 2-5 分钟的小动作等待额外评审；
- reviewer 不必在实现中途高频打断，只在阶段性结果出来后 review；
- 协作评论仍保留，但以阶段性广播为主，而不是逐动作广播。

### 3.2 白名单直接接管

若发现的问题满足“边界清晰、影响局部、修复确定性高”，另一代理可直接接手修复，并在修复后广播结果，而不必先等待原 owner 执行。

推荐纳入白名单的直接接管项包括：

- 错误导出或错误符号名；
- 伪测试、无效断言、无法真实失败的测试；
- 相对导入路径修正；
- `NodeNext` / `ESM` 所需的 `.js` 扩展名补齐；
- 显式类型补齐与局部类型错误修正；
- 构建依赖与类型声明补齐；
- 不改变业务语义的小范围 lint / build 修复。

直接接管的限制如下：

- 不得借机扩 scope；
- 不得修改冻结契约；
- 不得擅自更改共享入口的对外语义；
- 不得在未广播的情况下跨出原任务声明目录。

### 3.3 关键节点强制同步

以下场景仍然必须保持 fail-closed 的强同步：

- 修改共享文件；
- 修改冻结契约或共享协议；
- 调整目录 ownership；
- 变更分支策略或里程碑收口策略；
- 准备合入 `main`；
- 发现任务已从局部修复升级为架构性调整。

### 3.4 阶段 review 代替逐步 review

review 的默认颗粒度应从“每个提交”提升到“每个阶段包”。

推荐最小阶段包：

- `设计 / 计划` 阶段包；
- `实现 / 测试` 阶段包；
- `收口 / 合并` 阶段包。

只有当 reviewer 发现阻断性问题，才应把任务回退到前一阶段。

### 3.5 核心工作区协作（7-ARTIFACT-HUB-V2）

为提高核对效率与降低分支协作摩擦，后续协作以 `7-ARTIFACT-HUB-V2` 作为核心工作区：

- 默认占用范围为 `7-ARTIFACT-HUB-V2/**`
- 每个 AGENT 自建 `agent/*` 分支并提交独立 PR；治理只对合入 `main` 负责
- 允许少量例外文件用于支撑 `7-ARTIFACT-HUB-V2` 的 build/test/运行，但必须在 `STARTED/UPDATED` 中显式声明例外范围
  - 推荐允许范围：`docs/superpowers/contracts/7-artifact-hub-v2-foundation/**`
  - 其它例外仅限必要的 CI/模板/README 调整

## 4. 执行模式

### 4.1 模式一：默认并行模式

适用条件：

- 目录 owner 已明确；
- 修改范围不涉及共享文件；
- 契约已冻结；
- 当前任务属于局部实现、局部测试或局部文档补充。

执行规则：

- owner 运行门禁并发一次 `STARTED`；
- 在 owner 目录内直接连续推进；
- 范围扩大时才发 `UPDATED`；
- 阶段完成时统一发 `TEST_REPORT` / `DONE`。

### 4.2 模式二：直接接管模式

适用条件：

- 问题属于白名单直接接管项；
- 修复不改变任务边界与业务语义；
- 用户已预授权，或当前任务上下文已有明确接管许可。

执行规则：

- 接管方在评论中说明“接力修复 / direct takeover”；
- 接管方只修改声明范围内的局部文件；
- 修复完成后发 `DONE`，并明确 owner 应基于最新 PR head 继续。

### 4.3 模式三：强同步收口模式

适用条件：

- 修改共享文件或冻结契约；
- 准备合入里程碑或主分支；
- 需要重新分配 owner 边界；
- 需要统一判定“是否完成收口”。

执行规则：

- 必须先补 `DESIGN_REVIEW` 或 `UPDATED`；
- 必须统一跑 build / test / lifecycle checks；
- 必须由非主责 AGENT 给出收口结论；
- 未达到证据闭环前，不得宣告整体完成。

## 5. 评论与广播策略

本模式不取消结构化评论，但调整其频率与意图。

### 5.1 必须保留

以下评论仍为强制项：

- `STARTED`
- `UPDATED`
- `BLOCKED`
- `DONE`
- `DESIGN_REVIEW`
- `TEST_REPORT`

### 5.2 频率调整

推荐改为：

- 每个阶段至少一次 `STARTED`；
- 仅在范围变化时发 `UPDATED`；
- 仅在真正阻塞时发 `BLOCKED`；
- 每个阶段包结束时发一次 `DONE`；
- 不要求为每个微小提交单独评论。

### 5.3 评论内容调整

评论重点从“逐动作记录”调整为“阶段摘要 + 占用范围 + 下一步”。

推荐保留字段：

- 当前任务；
- 分支；
- 当前占用范围；
- 是否直接接管；
- 当前阶段结果；
- 下一阶段建议。

## 6. 目录 ownership 与共享边界

高效协作模式的前提，是目录边界比评论边界更清楚。

推荐规则：

- 优先按目录授予 owner，而不是按单文件逐个声明；
- owner 目录内默认允许连续推进；
- 共享入口文件单独列为共享边界，而不是混在整个目录 owner 里；
- `gatekeeper_config.json` 必须与任务板、实施计划、README 中的边界描述保持一致。

## 7. 对现有生命周期的影响

本模式不替换现有 9 个阶段，而是改变阶段的默认执行策略。

具体变化如下：

- `Phase 1` 与 `Phase 2`：仍需存在，但不要求每个实现子步都回到设计评审；
- `Phase 3`：仍然必须运行门禁，但通过后默认进入并行开发；
- `Phase 4`：默认并行推进，只有边界变化才同步；
- `Phase 5`：测试以阶段包统一汇总；
- `Phase 6`：以阶段 review 为主，避免对每个微小提交重复评审；
- `Phase 7`：仅在整体收口时统一判断是否可合入；
- `Phase 8`：保留归档与交接职责。

## 8. 推荐落地顺序

建议按以下顺序落地：

1. 先把本执行模式写入 `AGENT协作工具/docs/`；
2. 在 `README.md` 与生命周期设计文档中增加入口；
3. 在 implementation plan 中补充“默认并行、白名单接管、关键节点同步”的落地任务；
4. 再逐步把 checker / GitHub Actions 从“强制高频评论”调整为“识别关键缺失项”；
5. 最后再根据真实协作数据微调白名单与同步阈值。

## 9. 一句话结论

高效协作模式的核心不是减少纪律，而是：

> 让并行成为默认，让接管成为低成本，让同步只发生在真正高风险的节点上。
